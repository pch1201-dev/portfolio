0x03
방울토마토 자동 수확 로봇 시스템
Project 01 — Overview & Full Architecture
수행 기간
2025.03 - 2025.12 (캡스톤 디자인)
주요 플랫폼
Raspberry Pi 5, STM32, Arduino Mega
사용 기술
YOLOv5, RGB-D Camera, UART Serial

3D 카메라로 토마토 위치를 찾고, 그 좌표를 라즈베리파이에서 계산해서 하위 MCU(STM32, 아두이노)로 넘겨 로봇팔이 실제로 수확 동작을 하게 만드는 프로젝트입니다. 비전 처리와 여러 보드 간 통신을 엮는 과정에서 생기는 자잘한 문제들을 하나씩 풀어가는 게 이 프로젝트의 핵심이었습니다.

담당 역할 (Responsibilities)
  • → 3D 비전 및 로봇팔 제어 (메인 담당): Orbbec Astra 3D 뎁스 카메라로 X, Y, Z 좌표를 뽑아내고, 아두이노 메가로 로봇팔의 수확 궤적을 계산·제어하는 부분을 맡았습니다.
  • → 센서 데이터 안정화: 환경 노이즈로 뎁스 값이 흔들리는 문제가 있어서, 중간값 필터(Median Filter)를 적용해 해결했습니다.
  • → 멀티 MCU 통신 구조 설계: 라즈베리파이 5에서 계산한 좌표와 상태 값을 STM32, 아두이노로 지연 없이 넘기는 UART 통신 체계를 만들었습니다.
  • → 주행팀과의 협업: 비전에서 타겟을 인식하면 STM32 쪽 모터가 바로 정지하도록, 주행 담당 팀원과 신호 규격을 맞추고 통합 테스트를 함께 진행했습니다.
SYSTEM ARCHITECTURE
주요 기능 (Key Features)
FEAT_01
실시간 객체 탐지
YOLOv5와 OpenCV로 방울토마토를 실시간으로 감지하고 바운딩 박스를 그립니다.
FEAT_02
3차원 좌표 매핑
RGB-D 카메라의 깊이 데이터를 이용해 탐지된 타겟의 실제 3차원 좌표(X, Y, Z)를 계산합니다.
[DEBUG_LOG_SET: 방울토마토 수확 시스템 검증 및 최적화]
ISSUE_01 [RGB-D 카메라 토마토 인식 정확도 검증]
조도가 바뀌면 덜 익은 토마토까지 같이 인식되는 문제
야외 조명이나 날씨에 따라 색이 미묘하게 달라지다 보니, 잘 익은 토마토뿐 아니라 아직 덜 익은 것까지 타겟으로 잡히는 경우가 있었습니다.
RESOLVE
여러 조도 환경에서 데이터를 더 모아 YOLOv5를 추가로 학습시켰습니다. 이후로는 조명이 바뀌어도 잘 익은 토마토만 안정적으로 구분해냈습니다.
ISSUE_02 [로봇 주행 후 정지 타이밍 검증 (STM32)]
정지 신호를 받고도 로봇이 밀려서 멈추는 문제
라즈베리파이가 UART로 정지 명령을 보내도 STM32의 감속 응답이 살짝 늦고, 관성 때문에 바퀴가 밀리면서 목표 지점을 넘어서는 경우가 있었습니다.
RESOLVE
주행 담당 팀원과 함께 STM32의 정지 인터럽트 조건을 더 민감하게 조정하고, 속도 자체를 낮춰 관성의 영향을 줄였습니다. 이후로는 인식된 위치에서 오차 없이 멈추게 됐습니다.
ISSUE_03 [로봇팔 수확 동작 (역기구학) 정확도 검증]
손목 각도가 조금씩 어긋나 토마토 중심을 못 잡는 문제
좌표를 받아 역기구학으로 계산한 궤적대로 팔을 뻗어도, 링키지의 조립 공차나 서보 특성 때문에 손목 각도가 미세하게 어긋나 중심을 정확히 못 잡는 경우가 있었습니다.
RESOLVE
각 관절에 오프셋 보정값을 넣어 이 오차를 소프트웨어적으로 상쇄했습니다. 반복 테스트를 거치며 값을 조금씩 다듬었더니 수확 성공률이 눈에 띄게 올라갔습니다.
ISSUE_04 [3D 좌표계 기준 정의 및 중심 좌표 계산 정확도 검증]
좌표계 기준이 애매해서 중심 좌표가 자꾸 어긋나는 문제
처음에는 월드 좌표계와 카메라 픽셀 평면을 어떻게 맞출지 기준이 명확하지 않아서, 토마토 중심의 실제 좌표(X, Y, Z)를 계산할 때마다 값이 조금씩 어긋났습니다.
RESOLVE
화면 중앙 픽셀을 (0,0) 기준점으로 다시 정의하고, 카메라를 로봇팔 그리퍼 위로 옮겨 달아서(Eye-in-Hand) 시차 문제 자체를 줄였습니다. 이 두 가지를 같이 적용하고 나서야 좌표 값이 안정적으로 나왔습니다.

0x04
LiDAR 기반 자율주행 격자 지도 생성 시스템
Project 02 — Technical Log
수행 기간
2024.12 - 2026.02 (동아리 프로젝트)
주요 플랫폼
STM32F407VET6, HC-05 Bluetooth Module
사용 기술
LiDAR Sensor, A* Algorithm, Timer PWM, TFT LCD

라이다 센서로 주변을 스캔해 격자 지도를 만들고, 그 위에서 A* 알고리즘으로 최단 경로를 찾아 움직이는 자율주행 플랫폼입니다. 메모리와 연산 자원이 넉넉하지 않은 MCU 환경에서 이 알고리즘을 돌아가게 만드는 게 관건이었습니다.

담당 역할 (Responsibilities)
  • → 시스템 제어팀 팀장: STM32F407VET6와 HC-05로 휴대폰-보드 간 블루투스 통신을 구축하고, DC 모터 4개를 제어하면서 속도와 엔코더 값을 휴대폰 화면에 실시간으로 띄웠습니다.
  • → A* 알고리즘 구현: 방학 동안 A*를 공부해서 TFT LCD에 70×70 격자 지도를 구성하고, 랜덤 미로에서 (1,1)부터 (69,69)까지 최단 경로를 찾는 로직을 직접 구현했습니다.
  • → 라이다 기반 격자 지도 생성: 센서팀에서 받은 2D 라이다 데이터를 확률 기반 격자 지도(Occupancy Grid Map)로 변환하는 로직을 만들었습니다.
주요 기능 (Key Features)
FEAT_01
A* 경로 탐색
메모리 사용량을 줄인 최적화 버전으로 구현해, 랜덤 미로와 실측 맵 모두에서 최단 경로 탐색을 검증했습니다.
FEAT_02
Timer PWM 제어
하드웨어 타이머로 4개 DC 모터를 안정적으로 제어하고, 속도·엔코더 값을 무선으로 모니터링했습니다.
ISSUE [LiDAR 수집 데이터 왜곡]
라이다 버퍼가 밀려서 이상한 값이 튀는 문제
STM32가 UART로 라이다의 고속 데이터를 받는 도중에 연산 주기가 살짝 밀리면서 수신 버퍼가 넘치는 경우가 있었습니다. 그럴 때마다 말도 안 되게 먼 거리값이 섞여 들어와서 격자 지도에 이상한 노이즈가 생겼습니다.
RESOLVE
PC와 보드를 시리얼로 연결해서 원본 데이터를 그대로 찍어보며 어느 시점에 문제가 생기는지 추적했습니다. 특정 구간에서 버퍼가 제대로 비워지지 않는다는 걸 확인한 뒤, 범위를 벗어난 값은 걸러내고 그 자리를 앞뒤 프레임 값의 평균으로 채워 넣는 필터를 추가했습니다. 이후로는 지도에 튀는 값 없이 깔끔하게 나왔습니다.
0x05
스마트 강아지집 제어 시스템
Project 03 — Technical Log
수행 기간
2026.03 - 2026.06 (수업 과제)
주요 플랫폼
Arduino (ATmega2560 기반)
사용 기술
LM35 Temperature Sensor, Servo Motors, Bluetooth

온도 변화에 따라 반려동물의 생활 환경을 관리하는 아두이노 기반 제어 보드입니다. 센서 입력, 액추에이터 구동, 무선 통신을 처음부터 끝까지 직접 엮어보며 임베디드 기본기를 다지려고 진행한 수업 과제였습니다.

담당 역할 (Responsibilities)
  • → 하드웨어 제어 및 펌웨어 구현 (메인): 온도 센서 입력, 블루투스 통신, 자동 급식·급수용 서보 제어까지 메인 로직을 맡았습니다.
  • → 센서 값 안정화: 온도 값이 노이즈로 흔들리는 문제를 막기 위한 필터링과 예외 처리를 넣었습니다.
주요 기능 (Key Features)
FEAT_01
자동 온도 제어
온도 센서 값을 상시 확인하다가 설정한 임계치를 넘으면 팬을 자동으로 구동합니다.
FEAT_02
무선 상태 확인
블루투스 모듈을 통해 모바일 앱에서 실시간 온도와 팬 동작 상태를 확인할 수 있습니다.
ISSUE [스마트 강아지집]
온도 경계값 근처에서 팬이 껐다 켰다를 반복하는 문제
온도가 기준값 부근에 머무를 때 팬이 짧은 주기로 켜졌다 꺼지는 현상이 있었습니다. 팬이 꺼지자마자 온도가 다시 살짝 오르면서 곧바로 재작동 조건을 만족해버렸고, 이 상태가 계속되면 모터에 부담이 갈 수 있는 문제였습니다.
RESOLVE
기준값을 하나로만 두면 이런 문제가 생긴다는 걸 확인하고, LM35 센서 자체가 1℃당 10mV라는 작은 전압 차로 온도를 표현해 노이즈에 약하다는 점도 함께 고려했습니다. 그래서 켜지는 온도(28.0℃)와 꺼지는 온도(26.0℃)를 다르게 두는 히스테리시스(Hysteresis) 제어로 바꿨습니다. 2.0℃ 정도 여유를 두니 팬이 반복적으로 켜졌다 꺼지는 현상이 사라졌습니다.

배운 점
Conclusion
📌 Capstone Design Project

초반에는 "이걸 우리가 정말 해낼 수 있을까" 싶었던 순간도 많았습니다. 그래도 팀원들과 끝까지 붙잡고 늘어진 끝에 결과물을 완성할 수 있었고, 그 과정에서 협업이 왜 중요한지 몸으로 배웠습니다. 문제가 한꺼번에 몰아칠 때는 당황하지 않고 우선순위를 정해 하나씩 풀어나가는 게 결국 제일 빠른 길이라는 것도 알게 됐습니다. 이렇게 완성한 결과물로 교내 캡스톤디자인 경진대회에서 장려상을 받으면서, 그동안의 시행착오가 헛되지 않았다는 걸 확인할 수 있었습니다.

📌 Club Project (시스템 제어팀 팀장)

시스템 제어팀 팀장을 맡으면서, 팀원들이 겪는 어려움을 먼저 듣고 각자에게 맞는 방식으로 소통하려고 노력했습니다. 다른 팀들과 계속 맞춰가면서, 목표가 같아도 팀마다 일하는 방식이 다를 수 있다는 걸 배웠고 그걸 조율하는 법도 배웠습니다.

📌 Major Assignment Project

규모는 작았지만, 센서를 직접 연결하고 하드웨어를 하나하나 제어해보면서 임베디드 시스템의 기본기를 탄탄히 다질 수 있었던 프로젝트였습니다.
